Meet App

A Case Study in Serverless Architecture and Test-Driven Development

Project Pitch:

As part of my CareerFoundry training, I was tasked with building a serverless, progressive web application (PWA). The goal was to fetch and display real-world event data from a Google Calendar API, allowing users to find tech events in different cities. I developed the "Meet App" to not only meet these requirements but to rigorously apply Test-Driven Development (TDD) principles from the ground up. This project showcases my ability to integrate third-party APIs, implement a serverless backend, and build a reliable, offline-capable PWA.

The Process & Challenges

The core of this project was not just to build a functional app, but to build it using a specific, professional methodology (TDD) and a modern, secure architecture (serverless).

Challenge 1: Architecture: Implementing a Secure, Serverless API

Problem: The app needed to access the Google Calendar API, which requires secure authentication (OAuth2) and must not expose sensitive credentials (like API keys) on the client-side.

Solution: I engineered a serverless backend using AWS Lambda functions. This approach allowed me to create a secure API endpoint that handles the OAuth2 authentication flow and data fetching on the server side, protecting my API keys. The React client only communicates with my secure Lambda function, not directly with the Google API, which is a best practice for security.

Challenge 2: Development Methodology: Adhering to Strict TDD

Problem: The course required building the app using Test-Driven Development, which means writing tests before writing the application code. This was a new workflow.

Solution: I adopted a strict TDD workflow using Jest and React Testing Library. Every component (like CitySearch) and feature (like filtering events by city) was developed by first writing a failing test, then writing the code to pass it, and finally refactoring. This was a critical lesson in building resilient, predictable code and ensuring all features worked as expected from the start.

Challenge 3: Functionality: Offline Capability & Data Visualization

Problem: As a Progressive Web App (PWA), the app needed to be installable and functional even without an internet connection. It also needed to visualize the event data.

Solution: I implemented a service worker to cache the application shell (the HTML, CSS, and JS) and the event data fetched from the API. This allows users to launch the app while offline and still see the last-fetched event data. I also integrated Recharts to create scatter and pie charts, visualizing the number of events per city and by topic.

The Final Solution

The result is a fully functional, installable PWA that provides real-world event data. It features data visualization, a robust city search that filters events, and full offline capabilities, all built on a secure, serverless architecture.

Key Learnings & Future Steps

Key Learnings

This project was a deep dive into the "test-first" mindset, which fundamentally changed how I approach component development. It also solidified my understanding of serverless architecture (AWS Lambda) as a powerful, secure, and scalable solution for modern web applications.

Future Steps
